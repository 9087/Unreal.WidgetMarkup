2. XML Syntax and Properties

How element names map to UMG classes, how attribute names map to property paths, and how strings are converted into native values.

2.1 The element model

Every element in a markup file is the name of a Unreal class, looked up by its short class name (with prefix completion). The element type determines what kind of node the XML builder creates:

ElementMaps toBehaviour
<WidgetBlueprint>UWidgetBlueprintRoot container; compiled at End.
<WidgetTree>UWidgetTreeSets the blueprint's RootWidget.
<CanvasPanel>, <Overlay>, …Subclasses of UPanelWidgetAccept multiple child widgets.
<Border>, <Button>, …Subclasses of UContentWidgetAccept exactly one child widget.
<TextBlock>, <Image>, …Plain UWidgetLeaf widget.

The factory falls back to a property element when an unknown element is encountered while a parent widget is being built. This is what enables nested struct properties, see § 2.4.

2.2 Attributes are property paths

XML attributes set properties on the current element. Names use a dotted & bracketed canonical syntax:

  • Property — a top-level property of the element.
  • Outer.Inner — a sub-property of a struct.
  • Slots[0].Content — an indexed entry of an array.
  • Slot.LayoutData.Alignment — the layout slot of the parent panel.

Example:

<TextBlock
    Text="Hello"
    ColorAndOpacity.SpecifiedColor="1,0,0,1"
    Slot.LayoutData.Anchors.Minimum="0.5,0.5"
    Slot.LayoutData.Alignment="0.5,0.5" />

2.3 Value conversion

Strings are converted to native types by the ConverterRegistry. The built-in converters cover the property types commonly used in UMG:

CategoryExamples
NumericByte, Int32, Int64, Float, Double
BooleanTrue / False (case-insensitive)
Text-likeFString, FText, FName
EnumsAny UENUM, including byte-backed enums
Common structsFColor, FLinearColor, FMargin, FSlateChildSize, FSlateColor, FVector, FVector2D

Vector and color types accept comma-separated component lists, for example "1,0,0,1" for an opaque red FLinearColor.

2.4 Struct properties as child elements

For complex sub-properties you can also write them as child elements instead of using a long dotted attribute. The builder creates an intermediate property element that targets the right field on the parent: 

<Button>
  <Slot.LayoutData>
    <Anchors Minimum="0.5,0.5" Maximum="0.5,0.5" />
    <Alignment>0.5,0.5</Alignment>
  </Slot.LayoutData>
</Button>

The two forms are equivalent. Use whichever keeps a given block of XML readable.

2.5 The Name attribute

Every element accepts a built-in Name="..." attribute, registered at the UObject level so it works on any node. Effects:

  1. The target object is renamed to Name.
  2. If the target is a UWidget, it is also marked bIsVariable = true, so the UMG compiler exposes a blueprint property of the same name on the generated class. That lets you access the widget directly from blueprint or C++ code.
Names must follow blueprint variable rules: no ., no spaces, and they must be unique within the same blueprint. Duplicate names cause the compile to fail.

2.6 Wildcard property paths (advanced)

Custom property handlers may register paths with a wildcard array index, such as Slots[*].Content. The path matcher walks the inheritance chain of the target and picks the most specific match. You normally don't need to think about this; it just means that custom behaviour written for a base type also applies to subclasses.
